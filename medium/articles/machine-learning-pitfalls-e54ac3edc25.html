Machine learning pitfalls

Your ML project is as strong as its weakest link

Level 0 Problem definition

Real-world problems rarely manifest themselves as tractable data science problems. Therefore for any ML project, the very first step is to formulate the problem, in other words converting a high-level goal into a well-defined data science problem.

The greatest threat at this level is to come up with a problem definition that when solved wonât actually help anyone. Two examples from this article:

Level 1 Your first ML model

This level is about making an ML model work on a test set resting on your laptop. This is no easy task, but most of the content out there already focus on this point therefore I will only mention one pitfall:

Pitfall 1.1 Assuming more data will solve all of your problems

Irrelevant features or low-quality data will decrease the upper bound performance you can reach, which cannot be addressed with more data.

Underfitting due to low model capacity is another case where more data wonât help.

Level 2 Generalization

This is where it gets interesting. Say you have a model that works wonders on your test set, but shows a sharp performance decrease when tested with an external dataset. The following pitfalls might explain why:

Pitfall 2.1 Spurious correlations

If your dataset contains anything other than the signal (i.e. noise) that is correlated with your labels, your model will learn to exploit it for prediction.

In the example below the neural network classifies the image as a âhorseâ looking at the text, not the horse. Once the text is removed, the image is not classified as a âhorseâ.

Pitfall 2.2 Unrepresentative dataset

The sample you use to train your ML model should be representative of the population you are planning to use it on.

Say you are building a particle size predictor based on light scattering data, and you only trained it with spherical particles. If in practice people will use it with non-spherical particles as well, model performance will drop significantly.

Pitfall 2.3 Data mismatch between test and use-case

ImageNet is a database containing millions of images and thousands of classes. Year-over-year researchers developed better performing models on ImageNet challenge which lead to some people saying âVision is a solved problemâ.

Yet when tested on ObjectNet, the performance of these algorithms dropped around 40%. The reason is ObjectNet has proper controls, objects at all possible rotations backgrounds, and viewpoints. This is something ImageNet is lacking. Take a look at the sample dataset to see the differences.

Pitfall 2.4 Train test leakage

Letâs say you are working on a certain disease detection from tomography images. You will probably have multiple images belonging to the same patient. If you randomly split all your images into train and test sets, some images from the same patient might go to the training set and others can go to the test set. In this case, your model trained on the training set might have high performance on the test set because your model recognizes the patient, not the disease.

Pitfall 2.5 Changing data generating distribution over time

A data generating process can shift with time. Mobile phone cameras 10 years ago are very different from today. Hence if a model was developed using a dataset of mobile phone images from 10 years ago, but tested today the performance will likely be different.

Most statistical and machine-learning algorithms assume that the data is a random sample drawn from a stationary distribution. Unfortunately, most of the large databases available for mining today violate this assumption. They were gathered over months or years, and the underlying processes generating them changed during this time, sometimes radically. Source.

Level 3 System-level performance

We develop machine learning models, but they are usually put into use as part of a system that is part of another system. Therefore it is not enough to validate your model, you should also validate your system. Pitfalls at this level occur due to not having a system approach.

Pitfall 3.1 Faulty inputs

A perfect model will generate wrong predictions if the inputs are corrupted. The anti-stall software (MCAS) that caused the two 737 Max aircrafts crash five months apart relied on a single sensor for input. Failure of one sensor resulted in the death of 346 people.

It has no redundancy: it takes input from just one AoA sensor at a time. That makes MCAS completely unable to cope with a sensor malfunction. It canât âsanity checkâ its data against a second sensor or switch to a backup if the original source fails. It just believes whatever data itâs given, even if that data is bad, which is what happened on Lion Air flight 610 and Ethiopian Airlines flight 302. Source.

Level 4 Performance is not the outcome

Performance is not the outcome. Acing the metrics that were set during the problem definition does not guarantee the desired outcome.

Pitfall 4.1 Hidden stratification

A cancer detection model with high overall performance might consistently miss a rare cancer subtype, this is called the hidden stratification problem. The overall performance hides the variability in sub-group performance.

Pitfall 4.2 Demographic bias

Machine learning models acting on human data are susceptible to demographic bias. Amazon realized their AI recruiting tool was biased against women even if the gender was not explicitly marked on the CVs.

Solution?

Here are some tips I use for my day to day work at Ipsumio.

Tip 0: Do not use ML if you donât have to

Multiple times I was asked for ML solutions for problems that donât need ML. You can avoid all of the above challenges by not using machine learning if you donât have to.

Similarly, if an ML model with 3 parameters solves the problem, do not use a neural network with thousands of parameters.

Tip 1: Build the project iteratively

Instead of perfecting project stages and advancing linearly, go back and forth. Deploy a proof of concept as fast as possible and go back to the drawing board.

Tip 6: Error analysis

A simple but powerful tip. Look at all your errors and try to see if there is any systematic error. Say you are developing a car classifier and realize that all of your false negatives belong to cars occluded by an object. Either you can note âthe model wonât work under occlusionâ or you can increase the occluded car images in your training data and try fixing the issue.

Tip 7: Test the model with external datasets

If you developed a model using a dataset coming from one hospital, consider testing your model with external datasets, in the following order: prospective data from the same hospital, data from a different hospital in the same healthcare system, data from a different hospital in a different healthcare systemâ¦

Tip 8: Test the system

It is not enough to test the model, test it as a system to see how it will react with the other parts of the system and its environment.

Tip 9: Check the subgroup performance

During labeling the data, let the domain experts define and annotate subsets as well. Then analyze your model performance in these subsets.

Tip 10: Create the dataset thoughtfully

Make sure that

  • the signal you are searching for is inside the data
  • it is representative of the population
  • if you are generating the dataset, mimic reality instead of creating a perfect dataset. (e.g. if in practice low-resolution microscopes are used do not create your dataset with an expensive microscope).

Tip 11: Utilize MLOps

As we started seeing more production ML, deploying and maintaining these systems draw more attention. Use MLOps principals in your workflow.
