5 Mistakes Every Data Scientist Should Avoid

Precautions to be taken in a Data Science Project

When a challenging Data Science project comes, we sometimes in excitement or in an attempt to make a perfect model, end up doing mistakes. It’s important to cross-check the work from time to time and be cautious about the work. In this blog, I will discuss, five such commonly made mistakes by Data Scientists.

1. Data Leakage

Data leakage is a scenario that occurs when a model is trained on information that will not be available to it at prediction time. The result could be a model performing really well in training and testing time but performance degrades drastically in real-life scenarios. The reason sometimes is accidentally sharing information between the test and training data-sets instead of keeping test-data unseen and untouched.

Some possible Data leakages are :

During Pre-Processing

When a dimension reduction is performed using PCA (Principal Component Analysis), we reduce the n-dimensional data to k-dimensions (k<n) by learning eigenvectors from the covariance matrix created using training-data. The training data is transformed into a k-dimensional space by projecting the n-dimensional data in the directions of first-k eigenvectors. Then we train the model.

During testing, the right approach is to transform the test data into k-dimensional space by projecting the n-dimensional test data in the directions of first-k eigenvectors learned from the previous step and then do the model prediction. But, if mistakenly, PCA is applied to the whole data (training + testing data), it will create unnecessary data leakage and model performance will get affected.

Similarly, when applying an algorithm like NMF (Non-Negative Matrix Factorisation), the training on how that two matrices are learned happens during training. At test time, just the learnings are used to transform test data into required matrices. But, if mistakenly, NMF is applied to the whole data (training + testing data), it will create unnecessary Data leakages. Read this very well written blog for more details on Data Leakage.

2. Too many redundant features

Having highly correlated features affects the model’s performance. This is also referred to as the Curse of Dimensionality. The main issues faced due to having too many redundant features are :

  1. Overfitting: The redundant features try to overfit the model and result in high variance. This also affects the model’s accuracy during test time.
  2. No clear picture of Feature Importance: It’s necessary to understand which predictors are contributing more to understand the feature importance. This information helps in Data Science Storytelling and ranking features. Having redundant features, the model behaves abnormally and a clear picture of feature importance is lost.
  3. Computational time: With the increase in feature space due to redundant features, the computation time goes up. In scenarios where we need fast results, this may cause a problem as many models run time goes quadratically with the number of features.

The issue of having too many redundant features can be avoided by:

  1. Add features incrementally: The main idea is to keep the model simple. Add features incrementally using techniques like Forward and Backward selections. We want only important features to be part of the model at the end.
  2. Check for Multicollinearity and drop redundant features: We can check the correlation of features and delete redundant features.
  3. Perform Dimensional Reduction Techniques: Dimension Reduction techniques like PCA, Autoencoders, NMF can be very useful in removing redundant information at times.
  4. Regularisation: Regularisation techniques penalizes the model whenever it tries to overfit. Read more about it here.

3. Features Unavailability in future

Many times to improve on training data we use features that may not be directly or instantly available during testing.

One such example that I have faced in my work is Weather data. If we are using weather data for forecasting applications, we should keep in mind, actual weather data for the future during forecasting won’t be available and the best we can get is the weather predictions. Actual weather data during training and available weather predictions during testing/live-run may have a lot of deviation degrading the model performance.

4. Wrong Data Structure choice and Bad Coding style

I have faced issues when the application slows down drastically due to the wrong Data structure choice.

If the application requires fast look up for presence of an element use Dictionary or Set in python and avoid using List.

Dictionary/Set is the right Data structure for application with fast search needs

Coming to the coding style, the best practice is to have a modularised code and good variable and function naming style. It adds to understandability, interpretability and incorporating new requirements and changes easier. Check this link for the best coding practices.

5. Reporting Model performance before validating results for multiple Test periods, Data and Demographics

Many times, Data Scientists end up reporting model performance after testing in limited data. The model needs to be tested in a good amount of testing data in production before revealing the model performance with confidence.

  1. If it’s a Forecasting project, test the model performance/prediction for a number of time periods.
  2. If it’s a Finance Marketing model, test the model performance for the various markets across different demographics.
  3. If it’s a Retail domain model, test the model performance across categories, departments, etc

Conclusion

With this blog post, I wanted to point out 5 common but critical mistakes, Data Scientists end up doing. A bit of caution and awareness through this blog can save you and hopefully make things a bit easier.
