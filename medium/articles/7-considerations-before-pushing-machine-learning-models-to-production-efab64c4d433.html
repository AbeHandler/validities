7 Considerations Before Pushing Machine Learning Models to Production

Quick sanity checks before hitting the production road

Being part of a company that values scalability, I daily see, as a data scientist, the challenges that come with putting AI-based solutions in production.

These challenges are numerous and cover a variety of aspects: modeling and system design, data engineering, resource management, SLA, etc.

I don’t pretend mastery in any of those fields. I do however know that implementing some software engineering principles and using the right tools helped me a lot in making my work reproducible and ready for production.

In this article, I’ll share with you 7 of the considerations I have in mind before productionizing my models.

Let’s talk about this 💬

1 — Deal with randomness

A deterministic code is something less to worry about

Machine learning models are not deterministic: if you train a neural network twice using the same data and the same hyperparameters, you won’t get the same model. The outputs of these two models on the same test data may look very similar, but they are not identical.

This difference is due to multiple reasons. A very common one is related to the way the models are trained in the first place. In fact, optimizers, or in layman’s terms, the algorithms behind the minimization of the cost function (i.e. the training of the model, basically) introduce randomness when they compute gradients before updating the weight values: this randomness comes from the sampling of a train set.

🔴 Even though randomness is an intrinsic part of neural networks and many other machine learning models, it’s an undesirable quality in some contexts, especially when reproducibility matters.

The solution: set the random seed 🌱

By setting the seed for all the packages that use randomness (NumPy, SciPy, Torch, TensorFlow), you can force your code to be deterministic.

2 — Log your models, metrics, and artifacts

Code is not the only asset that should be versioned

Keeping track of your machine learning experiments is key to reproducing them at any point in time.

You can sure do this the old way by saving your models in folders and your metrics in JSON or (even worse) Excel files but this makes it hard to collaborate with people and maintain a consistent data structure over time.

I’ve recently started to use MLflow to track my machine learning projects and I couldn't be less happy.

In MLflow terminology, each project can be considered as an experiment with multiple runs inside.

Each run corresponds to an iteration where you can log:

  • model hyperparameters and additional configuration
  • model weights and binary files
  • additional artifacts that result from the run: text files, images, log files, or anything you want

MLflow stores this data on a tracking server. This can be a local server that you launch on your personal computer (in this case, the storage system will be your filesystem) or a remote server you deploy on the cloud.

It makes a lot of sense to use a tracking server on the cloud because teams can then work together and push artifacts to the same central location.

3 — Create pipelines and schedule them

You gotta put your fancy python scripts somewhere

It’s one thing that your code runs correctly on your local computer: you just press a button or type a command and launch it.

In a production environment, however, your code is integrated into a pipeline. This pipeline must be scheduled to run periodically, to react to potential failover, to launch virtual machines to run your code and scale it, to display information to the users, raise alerts or notifications, etc.

Your code is only a small part of the whole machinery 🏭 .

There’s a popular solution to create pipelines and orchestrate them: Airflow.

Airflow approaches pipelines as Directed Acyclic Graphs (or DAGs) that are composed of tasks. Tasks are units of computations that execute a python code.

Airflow provides you with the syntax to rearrange your tasks in any order you want as long as they fit in a DAG (which means, no cycles, ever.)

It also gives flexibility to programmatically schedule tasks.

4 — Externalize configurations (e.g. paths)

or any other sensitive information

You should avoid hardcoding variables such as paths into your code.

We all did it once and it may look something like this:

import pandas as pddata = pd.read_csv("../../../../data.csv")

But it works on my computer!

The problem with this practice is that anyone who’ll run your code must place the file in the exact same location as you or face an OS error. This is not very practical and limits the reproducibility of your work.

Paths should be considered as configuration parameters and must therefore be placed in external config files or injected in environment variables or read from some other systems. By separating the logic from the configuration, your code is more modular and easier to maintain.

When your code runs in production, paths and configurations are rarely. hardcoded: they’re read or loaded from external systems of files.

One solution to load environment variables inside a specific project without polluting the global namespace is to use the python-dotenv library (pip install python-dotenv ).

To be able to use this package, you’ll first have to create a .env file and put it, for example, at the root of your project. This file will contain environment variables that you want to be injected into your project runtime. These environment variables can include paths, credentials, hostnames, etc.

API_KEY=my_secret_key
PORT=3030
OUTPUT_PATH=/opt/trained_models/

Then, inside your main script, add the following line:

from dotenv import load_dotenv

To make environment variables accessible:

from dotenv import load_dotenv
import osapi_key = os.environ["API_KEY"]

5 — Setup CI-CDs to automate workflows

git push — and wait for the magic

Before you deploy a project to production, you usually want to make sure it builds and deploys correctly on a staging environment while validating unit tests.

CI-CD workflows allow you to automate such processes (and more) when triggered by Git events such as a push on specific branches or pull requests.

I use CI-CD pipelines with Github Actions to automate:

  • Unit tests and code coverage
  • Sanity check of the code by running code formatters (e.g. Black) or static type checkers (mypy)
  • Building Python packages and storing their wheels on S3
  • Automatic deployment to a remote server

6 — Don’t trust user inputs, validate them before

Never trust users. Sometimes they simply don’t know what they’re doing and when they do, they may want to mess with your code. So be prepared.

When you unleash your code into production, you can expect anything. Literally.

If you expect an integer and receive a string instead, your code should not silently fail or throw an indecipherable error message that the user cannot understand.

When your code fails in this particular situation, it must indicate to the end-user that the input he just entered does not conform to what the program is expecting. The code might expect a specific type, a string that matches a regular expression, or a value in an interval. If it receives something inconsistent, the user must know it, explicitly.

This is what we refer to as data validation.

There are different ways of handling data validation. One particular library I enjoy using for this purpose is Pydantic: it gives you the flexibility of adding validation on each attribute of a class.

7 — Use templates to bootstrap your projects

I hate repetition and I always strive to automate tasks. One particular aspect that I didn’t know was automatable is structuring projects.

Organizing projects can be a tedious and repetitive task. When I start a new project and want it to have a consistent structure with my previous work, I have to go over older projects, copy-paste their structure into the new folder, remove unnecessary files, add new ones, rewrite config files, etc. This is not a particularly difficult task, but the time is worth spending on something else.

You can automate this process by using the Cookiecutter library.

Cookiecutter allows you to create templates that kickstart a project: this is very practical if some of your projects follow the same structure. All you have to do is call cookiecutter and pass it the URL of the template you’re interested in. Cookiecutter will ask you about some information that you can fill in from the terminal and once you’re done, your project is created.
